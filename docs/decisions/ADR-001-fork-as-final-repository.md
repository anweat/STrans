# ADR-001：以 LING-lab72/STrans 作为结题与后续开发仓库

## 状态

已接受（Accepted）

## 日期

2026-07-15

## 背景

项目目前存在两条已经分叉的实现线：

- `LING-lab72/STrans`：包含项目组参与调优的识别算法、车辆快速入场恢复、ByteTrack 参数调整和检测数据趋势仪表盘，实际识别效果更好。
- `anweat/STrans` 及本地 `STrans_remote_test` 工作区：包含更完整的自动化测试、结题文档、SQLite 持久化、道路掩膜、录制导出、媒体部署和道路建模等附加能力，但算法方案与实际识别效果不如 fork。

截至本决策记录建立时，两个分支在共同祖先后分别包含 2 个和 24 个独有提交。本地迁移源还存在尚未提交的测试、文档和功能实现，因此不能仅依赖远端提交历史完成迁移。

结题阶段需要一个唯一、可验证、可复现的仓库。如果继续在两条实现线上并行开发，会造成算法参数、测试结论、文档描述和最终演示版本不一致。

## 决策

将 [LING-lab72/STrans](https://github.com/LING-lab72/STrans) 设为：

1. 唯一的后续开发主线；
2. 唯一的结题交付仓库；
3. 算法识别效果和演示行为的基准实现；
4. 测试报告、设计文档、使用说明书和答辩材料的事实来源。

当前 `STrans_remote_test` 工作区改为只读的迁移来源。迁移完成后，不再在旧仓库开发新功能。

迁移采用“以 fork 为底稿、按能力增量移植”的方式。禁止用旧仓库整目录覆盖 fork，也禁止为了兼容旧测试而恢复识别效果较差的旧算法行为。

## 受保护的基线

以下实现以 fork 为准，迁移时必须进行人工合并，不得用旧仓库版本直接覆盖：

- `backend/app/services/local_model.py`
- `backend/data/bytetrack_sandtable.yaml`
- `frontend/src/main.jsx`
- `frontend/src/styles.css`
- `frontend/vite.config.js`

其中提交 `bfff2c8`（`Add analytics dashboard and refine traffic detection`）是当前识别与仪表盘基线。后续若调整该基线，必须提供固定视频集上的对比结果和回归测试。

## 迁移内容

从旧工作区迁移以下独有价值：

- 后端与前端自动化测试；
- SQLite 公共连接、分析记录、认证、白名单、摄像头和报告持久化；
- 道路语义掩膜、道路热力图和模型下载；
- 视频录制、压缩、渲染、评估和导出；
- MediaMTX、Caddy 和凭据保护部署；
- 道路逻辑建模工具及前端入口；
- 需求、设计、测试、使用说明、项目总结和答辩文档源文件。

缓存、临时输出、编辑器配置和可重新生成的中间产物不迁移。

## 被否决的方案

### 继续以旧仓库为主线

否决原因：旧仓库的算法方案和识别效果不如 fork，后续需要重复移植 fork 的核心改进，容易发生算法回退。

### 将旧仓库整体合并或覆盖到 fork

否决原因：`local_model.py`、跟踪参数、主界面和样式存在关键差异，整体覆盖会破坏已经验证的识别行为和仪表盘功能。

### 长期维护两个正式仓库

否决原因：测试、文档、演示和代码会继续分叉，无法保证结题材料对应实际交付版本。

## 后果

### 正面影响

- 算法效果、演示版本和结题材料具有唯一事实来源。
- 可以将旧仓库的成熟工程能力逐项迁移，而不牺牲 fork 的识别效果。
- 每一批迁移都可以独立测试、审查和回退。

### 成本与风险

- `frontend/src/main.jsx`、`frontend/src/styles.css` 和 `backend/app/main.py` 需要人工拆分和合并。
- 旧测试中的部分断言可能绑定旧算法行为，必须按 fork 基线修订。
- 文档中的算法参数、截图和流程图需要重新核对。
- 本地未提交内容必须先形成可追踪的迁移快照，否则可能遗漏。

## 执行与验证

具体文件映射、迁移批次、质量门槛和结题冻结条件见 [迁移与结题开发指南](../migration/fork-convergence-plan.md)。

本决策只有在出现新的、经过同一固定数据集验证且显著优于当前 fork 的实现时，才能由新的 ADR 替代；旧 ADR 不删除。
