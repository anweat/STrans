# STrans 迁移与结题开发指南

## 1. 文档目的

本文档用于指导项目从 `STrans_remote_test` 向 `LING-lab72/STrans` 收敛，并确保 fork 最终成为可运行、可测试、可复现、文档一致的结题仓库。

核心规则只有一条：**迁移工程能力，不回退 fork 的算法与识别效果。**

关联决策：[ADR-001：以 LING-lab72/STrans 作为结题与后续开发仓库](../decisions/ADR-001-fork-as-final-repository.md)。

## 2. 仓库角色

| 仓库/工作区 | 角色 | 允许的操作 |
|---|---|---|
| `LING-lab72/STrans` | 后续开发主线、算法基线、结题仓库 | 开发、测试、文档、发布 |
| `STrans_remote_test` | 迁移来源和历史参考 | 建立迁移快照、读取、对比；不再新增业务功能 |
| `anweat/STrans` | 上游历史来源 | 仅用于追溯提交和必要同步 |

迁移开始前，必须把 `STrans_remote_test` 中尚未提交的有效内容保存为独立迁移快照。不要假设远端仓库已经包含本地全部测试和结题资料。

## 4. 不得回退的算法与产品基线

### 4.1 后端识别基线

保留 fork 中的以下行为：

- 新车辆进入画面时，车辆数量能够立即上升；
- 车辆数量下降继续使用中值平滑，降低偶发漏检造成的闪烁；
- 流模式使用快速入场恢复；
- 低置信度、无稳定轨迹的目标只在画面边界作为入场候选；
- ByteTrack `track_buffer` 使用 fork 当前值，除非有固定数据集证明新值更优。

受保护文件：

- `backend/app/services/local_model.py`
- `backend/data/bytetrack_sandtable.yaml`

迁移旧仓库功能时，可以调用或扩展这些能力，但不能复制旧文件覆盖。

### 4.2 前端产品基线

必须保留：

- 检测数据趋势入口；
- 车辆、检测框、事件指标切换；
- 趋势图、汇总指标、摄像头占比和检测明细；
- fork 当前的页面结构和视觉层次。

受保护文件：

- `frontend/src/main.jsx`
- `frontend/src/styles.css`
- `frontend/vite.config.js`

认证、摄像头管理、道路建模等旧功能应作为组件重新接入，不能用旧版 `main.jsx` 或 `styles.css` 覆盖。

## 7. 质量门槛

以下是迁移期间的默认门槛；项目组可以在取得第一轮基线数据后收紧，但不能无记录地放宽。

| 指标 | 门槛 |
|---|---|
| 固定视频集漏检率 | 不得比 fork 基线增加超过 2 个百分点 |
| 固定视频集误检率 | 不得比 fork 基线增加超过 2 个百分点 |
| 新车辆入场检测延迟 P95 | 不得比 fork 基线增加超过 200 ms |
| ID 切换次数 | 不得比 fork 基线增加超过 10% |
| 平均推理 FPS | 不得比 fork 基线下降超过 10% |
| 后端测试 | 全部通过 |
| 前端测试 | 全部通过 |
| 结题演示关键路径 | 连续执行 3 次无阻断错误 |

如果目前缺少可靠标签，应先记录“人工复核基线”，再逐步补充机器可读标注，不能用“看起来差不多”替代验收。

## 8. 高风险冲突处理

### `backend/app/services/local_model.py`

只允许以 fork 文件为底稿手工增加调用点或适配层。任何阈值、跟踪或平滑修改必须单独提交，并附算法对比结果。

### `backend/app/main.py`

按服务逐个注册路由和依赖，不复制整个文件。每迁移一组 API，就补充对应接口或服务测试。

### `frontend/src/main.jsx` 与 `frontend/src/styles.css`

先把 fork 的趋势仪表盘拆成组件，再迁移旧功能。禁止使用大段复制覆盖。样式需要限定作用域，防止旧选择器改变趋势图和主布局。

### `backend/requirements.txt`

使用依赖并集并逐项验证版本，不直接选择任一仓库文件。特别检查 PyTorch、Ultralytics、Transformers、OpenCV 和数据库相关依赖的兼容性。

### `.gitignore`

合并忽略规则，确保数据库备份、模型权重、凭据、视频和构建产物不会误提交。

## 9. Git 实施规则

每个迁移批次使用短期分支和原子提交，建议命名：

```text
codex/migrate-tests
codex/migrate-persistence
codex/migrate-road-mask
codex/migrate-recording
codex/migrate-frontend-tools
codex/migrate-final-docs
```

每个提交只做一类事情：

```text
test: establish fork recognition regression baseline
feat: migrate persistent camera configuration
feat: add road mask service without changing detector behavior
docs: align final report with fork implementation
```

禁止：

- 把 24 个旧分支提交一次性合并进结题主线；
- 在同一提交中混合算法调参与工程迁移；
- 在没有测试结果时修改受保护基线；
- 将本地数据库、密码、密钥、视频原始数据或模型权重提交到仓库。

## 10. 每批迁移的验收清单

- [ ] 变更目的和来源清楚。
- [ ] 没有覆盖受保护基线。
- [ ] 新能力有测试或明确的验证步骤。
- [ ] 后端与前端现有测试通过。
- [ ] 固定视频集指标满足门槛。
- [ ] README 或相关模块文档已更新。
- [ ] 未提交凭据、数据库、视频、模型权重或生成缓存。
- [ ] 失败时可以通过单个提交回退。

## 11. 结题仓库完成定义

只有同时满足以下条件，fork 才可以标记为结题版本：

1. 所有必迁 P0/P1 能力已经迁移或有书面放弃理由；
2. 固定数据集识别指标不低于本指南门槛；
3. 自动化测试与演示关键路径通过；
4. 需求、设计、测试、使用说明和实际实现一致；
5. 在一台干净电脑上可以依据 README 完成安装和启动；
6. 结题标签对应唯一提交，演示环境也使用该提交；
7. 旧仓库不再承载新功能开发。
