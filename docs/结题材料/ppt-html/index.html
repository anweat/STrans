02 · 问题定义

四类任务并不独立：真实难点来自尺度、时序、场景与工程约束叠加

沙盘小目标与遮挡

车辆和车牌像素少、视角变化大，单帧检测与 OCR 容易波动。

车辆—轨迹—车牌关联

识别结果必须跨帧稳定，才能支持白名单和通行决策。

道路标线与阴影误报

高对比箭头、灯光和相机轻微抖动会被误认为道路异常。

算法结果无法直接验收

还需要权限、历史、告警、证据、调度和真实前端形成软件闭环。

验证测试 + 数据 + 页面
价值可运行、可追溯
Problem → Constraint → Solution → Evidence

06 · 车辆与车牌算法

从单帧目标框，演进为“检测—跟踪—OCR—决策”的时序链路

先召回，再解释

YOLO 检测与 ByteTrack 跟踪后，叠加道路 ROI、类别、面积和外观过滤。

用时序降低波动

视觉 ID 恢复、检测框平滑、轨迹记忆和车牌多帧稳定共同抑制抖动。

形成业务输出

车辆裁剪 OCR 与周期全帧 OCR 互补，再连接白名单、速度、密度与拥堵。

Vehicle · Track · Plate · Decision
讲述重点：40 秒。重点讲为什么需要时序记忆和双路 OCR，不逐个讲阈值。

08 · 道路异常算法

多证据解释道路变化，并公开保留“标线误报”失败案例

变化候选

道路 ROI 内结合帧差、背景模型和光流稳定性产生候选区域。

合法目标解释

先由 YOLO/跟踪结果解释车辆与行人，再用颜色、纹理和形状过滤。

当前缺陷 S2

高对比道路箭头仍可能形成候选；事件冷却和生命周期合并不足会造成重复告警。

正确表述：“道路异常候选 + 多帧确认 + 人工复核”，而非完全自动判定。
Candidate · Evidence · Review
讲述重点：40 秒。主动展示失败案例，并说明它如何驱动下一轮标线抑制和事件去重。

09 · 道路建模与空间分析

把像素坐标映射到道路语义，才能回答“在哪条路、哪个路口”

配置产物

节点组、车道、建筑物、摄像头标定导出为 road_logic_modeler.v1 JSON，经人工审核后供 RoadLogicService 使用。

当前边界

发布审核以人工流程为主；后续再补草稿、校验、发布和回滚状态。

Geometry → Road Semantics → Traffic State
讲述重点：35 秒。用“三步”讲清：可视化建模、JSON 配置、实时道路映射。
