# STrans 结题交付总览

## 1. 总览目的

本文件是最终提交、答辩准备和材料审阅的唯一入口。事实基线为：

1. 初始要求：仓库根目录 `软件工程实训II-包老师.pdf`；
2. 当前实现：本仓库源代码、CodeGraph 结构索引和 Git 提交历史；
3. 测试证据：主项目 81 项、道路建模工具 26 项，共 107 个自动化测试单元；
4. 真实效果：F 盘真实沙盘视频抽样、CUDA 离线验证、前端实时流和 Playwright 截图；
5. 交付边界：无完整人工真值时不宣称严格精度；道路异常按候选事件和人工复核表述。
6. 硬件边界：ESP32-CAM 方案已整体放弃，最终采集源仅包含现有 RTSP/MJPEG、手机、USB 和本地媒体能力。

早期 V1.0 文档和阶段性实验记录用于保留项目历史，不作为最终答辩事实口径。最终提交应优先使用本目录的 V2.0 和结题版材料。

正式归档版已同步为 LaTeX：`docs/结题材料/latex/` 保存可复现源码，`output/pdf/latex-final/` 保存五份独立报告和一份综合报告。综合报告按“需求—设计—使用—测试—总结”编排，共 89 页，适合统一审阅；独立报告适合按成果物分别提交。

## 5. 统一答辩问题链

> 初始任务要求多源视频、四项识别和系统管理；真实沙盘又带来小目标、遮挡、标线误报、时序波动和设备差异。因此项目采用“模型 + 时序 + 几何 + 业务规则”的复合路线，并以 React/FastAPI、SQLite、权限、告警、证据和调度形成软件闭环。最后通过 107 个自动化测试单元、真实沙盘 CUDA 数据和前端实时流验证完成度，同时公开真值不足、道路标线误报和事件重复等限制。

这段话是报告、PPT 和现场回答应保持一致的总叙事。

## 6. PPT 与演示分工

### 6.1 10 分钟报告

使用 `ppt-html/index.html` 的 16 页母版。净讲述约 8 分 20 秒，预留约 1 分 40 秒用于换页、6 秒识别视频和停顿。

报告重点：

- 前 4 页回答“为什么做、要求是什么、难点是什么”；
- 第 5-12 页回答“系统和算法如何解决”；
- 第 13-15 页回答“如何证明、怎样迭代、价值是什么”；
- 第 16 页主动说明限制并切换演示。

### 6.2 10 分钟演示

1. 登录并启动真实视频；
2. 展示车辆、跟踪、车牌、速度和拥堵；
3. 切换道路异常模式并说明候选与失败案例；
4. 展示道路建模、摄像头标定和空间映射；
5. 展示告警处置、证据、历史和自适应调度。

### 6.3 5 分钟提问

优先准备：为什么没有严格精度、道路异常为何仍误报、为什么放弃 ESP32-CAM、是否达到三物理端协同、107 个测试代表什么、道路建模如何进入实时算法、CUDA 对精度还是时延产生影响。

详细答题边界见 `14-初始要求追踪与答辩问题链.md`。
