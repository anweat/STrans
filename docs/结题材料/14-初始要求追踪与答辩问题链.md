# STrans 初始要求追踪与答辩问题链

## 1. 文档目的与事实口径

本文把课程初始要求、当前仓库实现、系统测试证据和答辩叙事串成一条可核验链路，供软件说明书、总结报告和答辩 PPT 共同引用。

需求基线为仓库根目录的《软件工程实训II-包老师.pdf》。该 PDF 共 11 页，本文采用 PDF 页码引用。当前实现与测试事实以 2026-07-14 的 `LING` 基线及工作区结题材料为准。

状态含义如下：

| 状态 | 含义 |
|---|---|
| 功能级满足 | 已有代码入口，并有自动化、页面或真实数据证据支撑 |
| 部分满足 | 主链路存在，但精度、场景覆盖或验收证据尚不完整 |
| 待核验 | 可能已有材料或实现，但缺少明确版本、运行记录或验收证据 |
| 未完成 | 当前证据明确不足，不能在答辩中表述为已交付 |

证据使用遵循四级原则：

1. 初始要求回答“为什么做”；
2. 代码和 CodeGraph 回答“做了什么、模块如何连接”；
3. 自动化与浏览器测试回答“软件行为是否可重复”；
4. F 盘真实沙盘视频回答“在真实场景中表现如何”。

测试通过率不等同于需求覆盖率或算法准确率。当前没有逐帧人工真值，因此不得宣称 Precision、Recall、F1 或整牌准确率已经达标。

## 2. 初始要求摘要

### 2.1 项目目标

课程要求模拟一个“云边端协同”的智慧交通视觉感知系统：以手机或其他视频源作为采集端，以计算机完成视频接收和 AI 分析，并由前端展示结果；场景优先选择学院 704 智慧交通沙盘，也可使用其他道路场景（第 5 页）。

该目标可归纳为三个层次：

- **采集与协同**：视频源、计算节点和展示端能够通过网络或本地接口协同；
- **交通感知**：完成车辆、车牌、禁停和道路异常等识别任务；
- **管理闭环**：支持设备、模型、资源、监控、统计与历史查询。

### 2.2 四项核心交通识别任务

初始要求第 6 页明确提出：

1. 实时车牌检测识别，并与本地白名单比较，输出是否开放通行的决策；课程不要求实际控制闸机；
2. 检测所有车辆，统计分区车辆密度和数量，并动态绘制拥堵热力图；
3. 跟踪进入禁停区域的车辆，计算停留时长，超过阈值触发告警；
4. 检测道路上不应存在的异常物体，及时告警并标注位置与受影响车道。

### 2.3 管理与展示要求

初始要求第 7 页进一步提出：

- 监控 CPU、GPU、内存和视频传输状态；
- 支持边缘设备注册和在线状态监控；
- 支持模型选择与参数配置；
- 展示闸机监控、拥堵热力图、禁停区域和道路监控；
- 查询车牌检测数、拥堵变化、违规停车数、道路异常数和位置分布。

### 2.4 成果物与验收节奏

第 8 页要求提交需求分析报告 V1.0/V2.0、系统设计方案 V1.0/V2.0、源代码、软件说明书、系统测试报告、工作日报和总结报告。

第 9-10 页规定验收评审由“10 分钟报告 + 10 分钟演示 + 5 分钟提问”构成，并要求在 2026-07-15 提交源代码、软件说明书、系统测试报告和总结报告，源代码与软件说明书还需上传 GitHub 或 Gitee。

第 11 页给出的评分权重为：需求分析准确性 20%、系统方案合理性 20%、系统实现完备性 20%、系统实现效果 26%、系统创新性 8%、团队合作高效性 4%、成果物可读性 2%。这意味着答辩应优先呈现“真实效果、需求闭环和方案合理性”，而不是堆叠页面截图。

## 5. 当前测试结论如何支撑验收

### 5.1 已形成的证据链

- 主项目自动化回归 81/81：后端 62、前端 19；
- 道路建模工具 Python 21/21（含桥接 6 项）、Node 5/5；合计 107 个测试/脚本单元全部通过；
- 管理员、普通用户和道路建模页共 17 个浏览器检查点通过；
- 13 段视频均匀抽取 65 帧，全部由 CUDA 环境成功完成推理；
- live12 多车短片连续 24 帧稳定输出 3 辆车；
- 前端真实流已闭合“启动 - 推理 - 展示 - 历史 - 告警”链路。

这些证据能够证明系统主要模块可运行和可重复，但不能证明算法精度、所有 API 权限组合或长时间稳定性达到生产标准。

### 5.2 必须主动说明的限制

1. 没有人工框、车牌和异常真值，不能计算严格精度指标；
2. 道路箭头仍会触发异常候选，属于已记录 S2 问题；
3. 同类事件逐帧入库，缺少按相机、类型和区域的冷却合并；
4. 禁停、真实道路热力映射和白名单联动仍缺专项真实视频回归；
5. 尚未完成 API 权限/错误码矩阵、断流恢复和 30-60 分钟长稳测试；
6. 当前三端协同能力主要由接口和部署结构证明，未完成三台独立物理设备的系统验收。

## 6. PPT 的“问题 - 约束 - 方案 - 验证 - 价值”叙事链

答辩不要按“前端页面 A、页面 B、页面 C”顺序展开。建议围绕一个总问题和五个子问题讲述：

> **总问题：如何把多源交通视频转化为稳定、可解释、可管理、可追溯的道路事件与交通状态？**

### 6.1 子问题一：异构视频怎样稳定进入同一分析链路？

| 环节 | 内容 |
|---|---|
| 问题 | 手机、网络摄像头、本地录像等来源协议、帧率和可用性不同 |
| 约束 | 课程周期短；演示现场网络可能波动；前端和算法不能绑定某一种设备 |
| 方案 | 以 CameraHub/VideoStreamService 抽象视频源，统一状态、帧缓存和 MJPEG 输出；通过 FastAPI 和浏览器分离计算与展示 |
| 验证 | 真实 F 盘录像在管理页新增、启动并进入 CUDA 模型流；摄像头持久化测试和页面检查通过 |
| 价值 | 识别算法只处理统一帧接口，演示可以在实时源和离线素材间切换 |

PPT 对应：项目挑战页、部署架构图、实时识别全景图。

### 6.2 子问题二：通用模型怎样适应沙盘小目标和低帧率？

| 环节 | 内容 |
|---|---|
| 问题 | 沙盘车辆小、视角差异大，单帧检测会抖动、重复或细分类错误 |
| 约束 | 缺少大规模沙盘标注集，不能依赖重新训练一个完美模型 |
| 方案 | YOLO 检测 + ByteTrack 身份维持 + ROI/远近视角策略 + 重复抑制 + 轨迹平滑；车牌加入车辆裁剪、全帧补充和多帧稳定 |
| 验证 | live12 连续 24 帧稳定 3 辆车；65 帧 CUDA 评测；live3 产生真实 OCR 文本 |
| 价值 | 从“每帧画框”升级为带轨迹、车牌和稳定统计的车辆状态 |

PPT 对应：车辆活动图、多车短片、车牌短片；页脚注明“OCR 准确率待人工真值评测”。

### 6.3 子问题三：像素坐标怎样变成可用于管理的道路语义？

| 环节 | 内容 |
|---|---|
| 问题 | 检测框只能说明画面里有什么，不能直接回答位于哪条车道、是否禁停、是否拥堵 |
| 约束 | 摄像机存在透视畸变，不同机位坐标不能直接比较 |
| 方案 | 道路建模工具生成拓扑与标定 JSON；单应映射将车辆接地点投影到道路平面；RoadLogicService 计算车道归属、停留和热力状态 |
| 验证 | 道路逻辑 6 项、热力图 6 项、建模工具及桥接测试通过；同源页面真实加载 |
| 价值 | 把视觉检测转换为车道、路口、禁停和拥堵等业务语义，支撑可配置场景 |

PPT 对应：道路建模流水线、建模页面截图、热力图页面。

### 6.4 子问题四：异常检测怎样兼顾可解释性与可信度？

| 环节 | 内容 |
|---|---|
| 问题 | 沙盘缺少异常标注数据，背景变化、道路标线和正常目标容易产生误报 |
| 约束 | 无法在短周期内训练覆盖所有异常的端到端模型 |
| 方案 | 道路 ROI + 背景差分 + 光流相机运动判断 + YOLO 合法目标解释 + 外观/时序规则，输出候选并进入人工复核 |
| 验证 | 真实视频可产生候选和告警；同时记录道路箭头误报与事件洪泛问题 |
| 价值 | 每一层过滤都可解释，失败案例可直接转化为下一轮标线抑制与事件生命周期需求 |

PPT 对应：异常活动图，右侧并列“有效候选/道路箭头误报”。这一页主动呈现边界，反而更能证明测试真实性。

### 6.5 子问题五：识别结果怎样形成可验证的软件工程闭环？

| 环节 | 内容 |
|---|---|
| 问题 | 单纯展示检测框无法支持历史查询、告警处置、责任追踪和系统验收 |
| 约束 | 系统需要在单机演示环境中低成本部署，同时保留权限和审计能力 |
| 方案 | FastAPI + React + SQLite；识别结果进入历史、告警、证据帧、SHA-256 清单、处置和审计；资源状态驱动自适应调度 |
| 验证 | 81/81 主项目自动化、17/17 页面检查点、真实流落库回查；CodeGraph 验证模块连接 |
| 价值 | 形成“采集 - 识别 - 分析 - 告警 - 处置 - 归档”的闭环，并为后续迭代保留证据 |

PPT 对应：组件架构图、数据闭环时序图、系统测试数字和历史/告警截图。

## 9. 建议预设的提问与回答边界

### Q1：为什么没有给出识别准确率？

当前 65 帧测试用于验证数据链路、样例输出和性能，没有逐帧人工真值，因而不能把输出数量当作准确率。我们保留了原始帧和结构化结果，下一步会建立车辆框、车牌和异常真值，再计算 Precision、Recall、F1 和整牌准确率。

### Q2：道路异常为什么还有误报？

真实视频发现道路箭头会形成高对比静态候选，说明当前背景/外观规则仍需加入道路标线先验。同时同类事件缺少冷却合并。系统目前将其定义为“候选检测 + 人工复核”，误报案例已经转化为明确的下一轮测试和算法需求。

### Q3：是否真正实现云边端协同？

系统已经把视频源、分析服务和浏览器展示解耦，并支持 RTSP/MJPEG、手机视频、本地文件和可选转发，具备分端部署结构。早期 ESP32-CAM 硬件采集路线已整体放弃，不属于最终节点或验收范围。当前系统测试主要在单台 Windows 主机完成，尚未用三台独立物理设备做长稳验收，因此不扩大表述。

### Q4：是否控制了真实闸机？

没有。课程要求明确写明无需控制实体闸机。系统完成的是车牌、白名单比较和放行/拦截软件决策闭环。

### Q5：107 个测试是否意味着系统 100% 完成？

不是。107 表示已执行测试/脚本单元全部通过。API 权限矩阵、断流恢复、长稳测试和若干真实算法专项仍是覆盖缺口，已经在测试报告和追踪矩阵中列出。

### Q6：道路建模工具与识别算法有什么关系？

建模工具生产道路拓扑、车道和摄像头标定 JSON；RoadLogicService 消费该配置，将车辆画面坐标转换成车道、路口、禁停和拥堵语义。它是算法的空间配置与解释层，不参与每一帧神经网络推理。
