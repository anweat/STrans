# STrans 算法服务接口说明

本文档用于说明前后端框架与算法推理服务之间的对接方式。当前项目采用“前后端与算法服务解耦”的方式：

- 前端负责多摄像头画面展示、热力图、事件日志、历史记录和导出入口。
- 后端负责摄像头拉流、MJPEG 转发、接口网关、算法结果接收和 SQLite 历史记录归档。
- 算法服务负责 YOLOv11、ByteTrack、车牌识别、障碍物识别、道路异常检测等模型推理。
- 模型切换和模型管理建议在算法服务侧完成，前端只配置算法服务地址并展示返回结果。

## 1. 后端已提供的接口

### 1.1 摄像头与视频流

| 方法 | 路径 | 说明 |
|---|---|---|
| GET | `/api/cameras` | 获取沙盘固定 RTSP 摄像头和手动视频源列表 |
| POST | `/api/cameras` | 新增自定义视频源 |
| POST | `/api/cameras/{camera_id}/start` | 启动指定摄像头 |
| POST | `/api/cameras/{camera_id}/stop` | 停止指定摄像头 |
| GET | `/api/cameras/status` | 获取全部摄像头运行状态 |
| GET | `/api/cameras/{camera_id}/status` | 获取指定摄像头状态 |
| GET | `/api/cameras/{camera_id}/mjpeg` | 获取前端展示用 MJPEG 视频流 |
| POST | `/api/video/start` | 手动接入手机 IP Webcam、本地视频或其他视频源 |
| GET | `/api/video/mjpeg` | 获取当前选中视频源的 MJPEG 视频流 |

### 1.2 算法服务配置与调用

| 方法 | 路径 | 说明 |
|---|---|---|
| GET | `/api/algorithm/config` | 获取当前算法服务配置 |
| PUT | `/api/algorithm/config` | 配置算法服务地址、健康检查路径和推理路径 |
| GET | `/api/algorithm/health` | 检查算法服务是否可用 |
| POST | `/api/algorithm/infer/{camera_id}` | 后端取当前摄像头一帧，转发给算法服务 `/infer` |
| POST | `/api/algorithm/results` | 算法服务主动推送检测、跟踪、统计和事件结果 |
| GET | `/api/analysis/latest` | 获取最新一条算法分析结果 |
| GET | `/api/analysis/events` | 获取近期事件日志 |

### 1.3 历史记录与导出

算法结果通过 `/api/algorithm/infer/{camera_id}` 返回或通过 `/api/algorithm/results` 推送后，后端会自动写入 SQLite 数据库。

| 方法 | 路径 | 说明 |
|---|---|---|
| GET | `/api/history?limit=30` | 获取历史分析记录 |
| GET | `/api/history?camera_id=live1&limit=30` | 按摄像头查询历史记录 |
| GET | `/api/history/export?format=csv` | 导出 CSV 报告 |
| GET | `/api/history/export?format=json` | 导出 JSON 报告 |

SQLite 默认路径：

```text
backend/data/traffic_analysis.db
```

## 2. 算法服务建议提供的接口

算法服务可使用 FastAPI 或其他 HTTP 服务实现。假设算法服务地址为：

```text
http://算法电脑IP:9000
```

前端或后端配置时填写该基础地址即可。

### 2.1 GET /health

用于健康检查。

返回示例：

```json
{
  "status": "ok",
  "models": ["yolov11", "bytetrack", "hyperlpr3", "s2m"]
}
```

### 2.2 POST /infer

当前后端点击“分析当前帧”时，会把当前摄像头最新一帧图片以 base64 JPEG 的形式发送给算法服务。

请求示例：

```json
{
  "camera_id": "live1",
  "image_base64": "base64 encoded jpeg",
  "config": {
    "confidence": 0.35,
    "iou": 0.45,
    "detection_interval_ms": 500,
    "inference_size": 640,
    "enabled_tasks": ["vehicle", "tracking", "plate", "obstacle", "traffic"]
  }
}
```

## 3. 算法服务主动推送结果

如果算法服务直接读取 RTSP 视频流并持续推理，也可以不走 `/infer`，而是主动把结果推送到前后端后端：

```text
POST http://前后端电脑IP:8000/api/algorithm/results
```

请求体格式与 `/infer` 返回格式保持一致。后端收到结果后会：

1. 更新最新分析结果；
2. 更新事件日志；
3. 写入 SQLite 历史记录；
4. 供前端大屏、热力图和历史记录页面展示。

## 4. 字段约定

| 字段 | 说明 |
|---|---|
| `camera_id` | 摄像头编号，对应 `/api/cameras` 中的 `camera_id`，如 `live1` |
| `source_width` / `source_height` | 原始视频帧尺寸，前端叠加检测框和热力图时需要 |
| `bbox` | 原始图像像素坐标，格式为 `[x1, y1, x2, y2]` |
| `class_name` | 检测类别，如 `car`、`person`、`obstacle` |
| `confidence` | 置信度，范围 `0-1` |
| `track_id` | ByteTrack 等跟踪算法维护的车辆 ID |
| `plate` | 车牌号或电子 ID，可为空 |
| `speed_kmh` | 估计速度，可为空 |
| `congestion_level` | `low`、`medium`、`high`、`unknown` |
| `severity` | `info`、`warning`、`critical` |

## 5. 推荐联调流程

1. 启动后端：`python -m uvicorn app.main:app --host 0.0.0.0 --port 8000`
2. 启动前端：`pnpm dev` 或 `npm run dev`
3. 前端打开 `http://localhost:5173`
4. 启动一路沙盘 RTSP 摄像头或手机视频源
5. 启动算法服务，例如 `http://192.168.1.20:9000`
6. 在前端填写算法服务地址并点击连接
7. 点击“分析当前帧”测试 `/infer` 链路
8. 若算法服务主动推送，则调用 `/api/algorithm/results`
9. 前端检查检测框、统计指标、事件日志、热力图和历史记录

## 6. 注意事项

- 前端不直接管理模型权重，也不负责模型切换。
- 如果需要切换 YOLOv11、ByteTrack、HyperLPR3、S2M 等模型，请在算法服务侧完成。
- 前端展示依赖 `source_width`、`source_height` 和 `bbox`，算法返回时不要省略。
- 历史记录只有在后端收到算法结果后才会写入 SQLite。
- 校园网沙盘 RTSP 摄像头需要电脑处于可访问摄像头的网络环境。
