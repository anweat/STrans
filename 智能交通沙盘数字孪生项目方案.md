# 基于多点视觉感知的智慧交通沙盘数字孪生与信号优化系统

> **历史方案说明：** 本文中的 ESP32-CAM 路线已由项目组整体放弃，仅保留为早期方案演进记录，不属于最终实现与结题交付范围。最终方案以 `docs/结题材料/17-需求分析报告V2.0.md` 和 `18-系统设计方案V2.0.md` 为准。

## 1. 项目题目

推荐题目：

**基于多点视觉感知的智慧交通沙盘数字孪生与信号优化系统**

可选题目：

- **面向交通沙盘的云边端协同智能感知与交通指挥优化系统**
- **基于 ESP32-CAM 与视觉定位的智慧交通数字孪生仿真平台**
- **基于多摄像头感知与交通流仿真的自适应信号灯控制系统**

## 2. 项目定位

本项目面向学校智慧交通沙盘场景，构建一个集多点摄像头采集、视觉感知、沙盘坐标映射、交通流 Mock 仿真、红绿灯控制、路径优化和可视化展示于一体的智慧交通系统。

由于沙盘车辆大概率是静态模型，系统不把视觉识别结果直接等同于真实交通流，而是采用“视觉感知 + 数字孪生 + Mock 交通流仿真”的方案：

```text
多点摄像头采集
  ↓
车辆 / 车牌 / 障碍物 / 道路区域识别
  ↓
ArUco / 二维码标定与沙盘坐标映射
  ↓
构建数字孪生交通场景
  ↓
Mock 车辆速度、路线、排队和拥堵演化
  ↓
红绿灯配时优化 / 路径规划 / 交通指挥策略
  ↓
三维或二维大屏展示 + 实体红绿灯联动
```

## 3. 核心问题与设计思路

### 3.1 沙盘车辆不移动的问题

沙盘车辆如果是静态摆放，视觉系统只能识别车辆位置、道路占用和异常物体，无法真实获得速度、排队增长和通行时间。

解决方案：

- 视觉模块负责获取初始状态和场景校准；
- Mock 仿真模块负责生成车辆运动、速度、路线和拥堵变化；
- 决策模块基于仿真状态进行红绿灯控制和路线优化；
- 展示端同时展示真实摄像头画面和数字孪生仿真结果。

### 3.2 多摄像头位置与标定问题

前期手机摄像头可能位置不固定，后续可使用 ESP32-CAM 或 USB 摄像头固定在夹具上。

推荐采用 ArUco / AprilTag 标记进行自动标定：

- 在沙盘关键位置贴标记；
- 摄像头画面识别标记 ID 和角点；
- 计算图像坐标到沙盘坐标的 Homography；
- 将车辆、障碍物、路口和道路区域映射到统一地图。

第一版建议使用 ArUco，因为 OpenCV 原生支持，开发成本低。

### 3.3 YOLO 是否是必要模型

YOLO 不是必须项。系统可以根据任务选择不同模型或传统视觉方法：

- 车辆检测：YOLO / RT-DETR / PP-YOLOE，也可先用颜色、边缘、模板或标记辅助；
- 车牌识别：更推荐“定位点/边框检测 + 透视矫正 + OCR/CRNN”；
- 道路异常：不应只依赖 YOLO，应采用“背景差分 + 区域规则 + 检测模型 + 时序判断”的混合方案；
- 交通决策：不依赖图像模型，而依赖交通流特征、规则模型或机器学习模型。

### 4.1 硬件层

已有硬件：

- ESP 开发板；
- ESP32-CAM 完整模组；
- STM32 开发板；
- 学校智慧交通沙盘；
- 电脑若干；

建议硬件分工：

```text

  负责固定点位视频采集与 MJPEG/HTTP 推流。

```

## 5. 视觉标定与沙盘映射方案

### 5.1 标记设计

在沙盘上贴 ArUco 标记，例如：

```text
ID 0：沙盘左上角
ID 1：沙盘右上角
ID 2：沙盘右下角
ID 3：沙盘左下角
```

或在每个路口、道路节点贴不同 ID 的标记。

### 5.2 坐标映射

系统读取摄像头画面后：

1. 检测 ArUco 标记；
2. 获取标记四角在图像中的像素坐标；
3. 根据预设沙盘坐标计算 Homography；
4. 将检测到的车辆、障碍物、禁停区等投影到统一沙盘地图。

### 5.3 为什么不是完整 3D 重建

严格 3D 重建需要多视角标定、深度估计、点云或 mesh 构建，开发风险较高。当前项目更适合采用“轻量数字孪生”：

- 用标记完成二维或伪三维坐标映射；
- 在前端用二维地图或 Three.js 展示三维沙盘；
- 车辆、道路、信号灯在数字场景中动态更新；
- 不做复杂的 SfM、SLAM、NeRF 或 Gaussian Splatting。

## 6. 车牌识别模块设计

### 6.1 设计目标

车牌识别模块用于实现车辆身份识别、白名单比对和闸机/通行策略控制。

输出字段：

```text
vehicle_id
plate_no
vehicle_type
whitelist_status
confidence
gate_action
```

### 6.2 推荐路线：定位码辅助车牌识别

由于沙盘车辆较小，传统车牌可能分辨率不足，建议设计“带定位点的工程化车牌”：

```text
[定位点] A12345 [定位点]
```

或：

```text
四角定位点 + 中间车牌字符
```

识别流程：

```text
摄像头画面
  ↓
检测车牌边框 / 四角定位点 / 小型 ArUco 标记
  ↓
透视矫正为标准车牌图
  ↓
灰度化、去噪、锐化、二值化
  ↓
OCR / CRNN / PaddleOCR / RNN-CTC 识别字符
  ↓
白名单 / 黑名单 / 权限策略判断
```

### 6.3 可选技术方案

方案 A：传统视觉 + OCR

- OpenCV 检测边框、角点或定位点；
- 透视变换裁剪车牌；
- PaddleOCR 或 CRNN 识别字符；
- 适合第一版实现。

方案 B：ArUco 电子车牌

- 每辆车贴一个小型 ArUco；
- ID 映射为车牌号；
- 识别稳定，适合沙盘演示；
- 可以作为 OCR 失败时的兜底。

方案 C：真实车牌检测 + OCR

- 使用车牌检测模型定位车牌；
- OCR 识别车牌；
- 更接近真实场景，但对画质要求高，沙盘上风险较大。

### 6.4 推荐实现

第一版采用：

```text
ArUco / 定位点辅助定位 + OCR 识别 + ID 兜底
```

这样既能展示“真实车牌识别”流程，也能保证沙盘演示稳定。

## 7. 道路异常检测模块设计

### 7.1 异常类型定义

道路异常不应只理解为“障碍物检测”。建议定义为多类交通异常事件：

```text
1. 道路障碍物
2. 行人 / 非机动车闯入
3. 车辆长时间停留 / 违停
4. 逆行
5. 禁行区闯入
6. 路段异常拥堵
7. 摄像头遮挡 / 偏移 / 画面质量异常
```

### 7.2 分层检测方案

推荐架构：

```text
固定摄像头画面
  ↓
ArUco 标定道路 ROI 和车道区域
  ↓
背景差分检测新增前景
  ↓
目标检测 / 分割模型识别已知类别
  ↓
跟踪与时序特征提取
  ↓
规则引擎 + 异常分类模型
  ↓
生成异常事件与处置策略
```

### 7.3 各类异常的具体解法

| 异常类型 | 推荐解法 |
|---|---|
| 道路障碍物 | 背景差分检测新增物体，判断是否位于车道 ROI 内 |
| 行人闯入 | 使用行人检测模型；沙盘小人可自采数据微调 |
| 车辆违停 | 目标跟踪 + 速度接近 0 + 停留时间超过阈值 |
| 逆行 | 车辆轨迹方向与车道允许方向相反 |
| 禁行区闯入 | 目标中心点进入禁行 ROI |
| 拥堵异常 | 车辆密度高、平均速度低、等待时间持续增长 |
| 摄像头异常 | 标记丢失、标记位置偏移、图像过暗、模糊或遮挡 |

### 7.4 模型选择建议

第一版：传统视觉 + 规则

- OpenCV 背景建模；
- ROI 区域判断；
- 轨迹和停留时间规则；
- 实现简单，适合固定沙盘场景。

第二版：已知类别检测模型

- YOLO11 / YOLOv8 / RT-DETR / PP-YOLOE；
- 识别车辆、行人、路障、交通锥、箱子等；
- 用于增强异常类型识别。

第三版：开放词汇或分割模型

- Grounding DINO / Grounded-SAM / SAM2；
- 用文字提示识别“road block”“person”“object on road”等；
- 适合低频复核或高级展示，不建议作为主实时链路。

### 7.5 推荐实现

第一阶段不把道路异常完全交给 YOLO，而是采用：

```text
背景差分 + 道路 ROI + 规则判断 + 少量检测模型
```

这样更符合固定摄像头沙盘场景，也更稳定。

## 8. Mock 交通流仿真模块

### 8.1 目标

Mock 模块用于弥补沙盘车辆静态的问题，在数字孪生场景中生成车辆运动、速度、排队、拥堵和通行结果。

### 8.2 输入

```text
道路拓扑
车辆初始位置
道路容量
红绿灯状态
障碍物状态
禁行区状态
摄像头检测结果
手动触发事件
```

### 8.3 输出

```text
车辆实时位置
车辆速度
排队长度
等待时间
路段拥堵等级
路口通行效率
推荐路线
信号灯控制策略
```

### 8.4 仿真规则

基础规则：

- 红灯时车辆停止或排队；
- 绿灯时车辆按速度通过；
- 车辆密度越高，路段速度越低；
- 障碍物出现时，道路容量下降或封闭；
- 路段拥堵时，路径规划提高该路段权重；
- 信号灯优化根据各方向排队长度和等待时间动态调整。

## 9. 交通状态模型设计

### 9.1 模型定位

为了体现“自研模型开发”，建议将模型放在交通状态判别和决策建议层，而不是从零训练图像检测模型。

模型输入为结构化交通特征：

```text
road_id
vehicle_count
avg_speed
queue_length
waiting_time
roi_occupancy
obstacle_flag
wrong_direction_count
signal_phase
camera_quality
```

模型输出：

```text
congestion_level
incident_type
road_weight
recommended_green_time
reroute_required
```

### 9.2 可选模型

第一版：

- RandomForest；
- XGBoost；
- LightGBM。

优点是数据量要求低、训练快、可解释性强。

第二版：

- LSTM；
- GRU；
- TCN。

适合基于最近 10 到 30 秒的时序特征判断拥堵趋势。

第三版：

- 强化学习 DQN / PPO；
- 多智能体信号灯控制；
- 可作为后续拓展，不建议第一阶段主线实现。

## 10. 红绿灯控制与路径优化

### 10.1 自适应红绿灯

第一版采用加权评分控制：

```text
score = 车辆数 * w1 + 排队长度 * w2 + 等待时间 * w3 + 拥堵等级 * w4
```

分数最高的方向优先获得绿灯，同时设置：

- 最小绿灯时间；
- 最大绿灯时间；
- 黄灯过渡时间；
- 防止某方向长期饥饿的等待补偿。

### 10.2 路径优化

将沙盘道路抽象为图：

```text
路口 = 节点
道路 = 边
边权重 = 通行时间 + 拥堵惩罚 + 异常惩罚 + 信号灯等待成本
```

路径算法：

- Dijkstra；
- A*；
- Floyd；
- 动态最短路径。

当检测到拥堵或异常：

- 提高对应路段边权重；
- 严重异常时封闭路段；
- 重新计算推荐路线；
- 前端显示新的交通指挥策略。

## 14. 项目创新点

1. **视觉感知与数字孪生结合**
   - 不是单纯目标检测，而是将检测结果映射到沙盘路网和数字孪生场景。

2. **面向静态沙盘的 Mock 交通流仿真**
   - 解决沙盘车辆不移动导致无法获得速度、排队和拥堵演化的问题。

3. **ArUco 辅助自动标定**
   - 支持摄像头固定点位快速映射，降低多摄像头部署难度。

4. **车牌识别采用定位码辅助方案**
   - 兼顾真实车牌 OCR 和沙盘演示稳定性。

5. **道路异常采用混合检测策略**
   - 结合背景差分、区域规则、检测模型和时序判断，不依赖单一模型。

6. **交通决策闭环**
   - 从感知、仿真、判断到红绿灯控制和路线优化形成完整闭环。
